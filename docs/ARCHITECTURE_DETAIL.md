# 集装修 - 技术架构说明

## 项目架构

本项目采用**云原生微服务架构**，按业务域拆分服务，支持独立迭代、弹性扩展。

## 技术栈

| 服务类型 | 语言 | 框架 | 用途 |
|----------|------|------|------|
| 高并发服务 | Go | Gin | 预算引擎、支付服务、用户服务 |
| AI服务 | Python | FastAPI | 验房、设计、施工、监工 |
| 前端 | TypeScript | React/Taro | 小程序、管理后台 |

## 微服务拆分

```
packages/
├── api/           # Python FastAPI - AI服务（验房、设计、施工、监工）
├── engine/        # Go Gin - 预算引擎（高并发、实时联动）
├── services/      # Go - 支付服务、用户服务（高并发）
├── web/           # 微信小程序（Taro）
└── admin/        # PC管理后台（React）
```

## 数据库设计

| 数据库 | 用途 |
|--------|------|
| PostgreSQL | 主业务库（用户、订单、设计方案） |
| Redis | 缓存（Token、会话、热点数据） |
| COS | 对象存储（效果图、施工图、照片） |

## AI能力层

```
用户请求 → AI中台适配层 → 模型路由 → 垂直引擎 → 人工兜底 → 返回
```

### 三大核心壁垒

| 功能 | 响应时间 | 技术方案 |
|------|----------|----------|
| AI验房 | ≤10秒 | 阿里云视觉 + 家装垂直微调 |
| AI设计+预算联动 | ≤300ms | 智谱GLM + 自研联动引擎 |
| AI云监工 | ≤15秒 | 阿里云视觉 + 进度识别引擎 |

## API网关

- **Nginx**：七层负载均衡、SSL终端
- **APISIX**：API网关、限流熔断

## 开发规范

1. 所有接口必须编写文档（使用 Apifox 管理）
2. 代码必须遵循行业规范（Go: GoLand规范, Python: PEP8）
3. 数据库操作必须使用参数化查询
4. 禁止硬编码密钥、敏感信息
5. 核心功能必须有单元测试
